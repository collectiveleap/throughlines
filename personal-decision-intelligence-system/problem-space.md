# Problem Space: Personal Decision Intelligence System

## Status
Working document. Derived from iterative design session. Not yet a specification.

---

## 1. Scope

A system ingesting communications and activity data belonging to a single individual, operating across multiple concurrent projects, with the goal of capturing facts, proposing actions, and accumulating a personal decision record over time.

---

## 2. Input

### 2.1 Known input types
- Emails
- Calendar entries
- Slack messages

### 2.2 Extensibility requirement
The input schema must accommodate source types not known at design time. Source type is a variable, not a fixed enumeration.

### 2.3 Ingestion properties
Each input must be:
- Timestamped at ingestion
- Tagged with source type
- Held as uninterpreted at ingestion

Interpretation is proposed by the system, not assigned at ingestion.

---

## 3. Fact Capture

### 3.1 Need
A way to represent what is currently true — about inputs, projects, relationships between entities, and states of affairs — in a form that is unambiguous, persistent, and queryable.

### 3.2 Constraints
- Facts must be separable from interpretations of facts
- Facts must be linkable to the inputs that generated them
- The fact representation must support the multi-project context (see Section 6)

---

## 4. Proposal Representation

### 4.1 Need
A way to represent candidate interpretations and candidate actions that the system proposes but which have not yet been approved by the human. Proposals must be distinguishable from facts about current state.

### 4.2 Proposal components
Every proposal consists of three parts, held separately:

1. **Candidate interpretation** — what this input or situation means
2. **Candidate action** — what should be done in response
3. **Candidate scope** — whether this is a narrow case or an instance of a generalizable rule

### 4.3 Scope deferral constraint
The candidate scope cannot be determined at proposal time. It can only be determined retrospectively, once sufficient similar cases have accumulated. The system must not require scope resolution at approval time.

---

## 5. Rule Genesis

### 5.1 Initial state
The system begins with zero rules. No policy structure exists. Every proposed action is therefore a candidate precedent, not an exception to an existing rule.

### 5.2 Approval semantics
When a human approves a proposed action in the zero-rules state, the approval is a legislative act — it potentially creates a rule — not an administrative act of permitting a deviation from existing policy.

### 5.3 Accumulation lifecycle
The system passes through phases:

- **Phase 1 — No precedent**: System proposes actions from facts alone, with no prior cases to draw on
- **Phase 2 — Sparse precedent**: System surfaces similar prior cases alongside proposals, for human reference
- **Phase 3 — Dense precedent**: System proposes candidate rules abstracted from accumulated cases, for human ratification

Transition between phases is not time-based but case-density-based. The threshold for transition is not yet specified (see Section 9).

### 5.4 Four-layer structure
The system must maintain four distinct layers:

1. **Raw facts** — ingested inputs, timestamped, source-tagged, uninterpreted
2. **Proposed interpretations** — candidate readings generated by the system, linked to the facts that generated them
3. **Approved cases** — human-ratified interpretations and actions, forming the case library
4. **Emergent patterns** — system-surfaced candidate rules, awaiting human ratification, linked to the cases they abstract over

### 5.5 Layer promotion constraint
Nothing may move from Layer 2 (proposed) to Layer 4 (emergent pattern) without passing through Layer 3 (approved cases). Generalization must always be grounded in approved cases, never inferred directly from raw facts.

---

## 6. Multi-Project Context

### 6.1 Competing claims
Multiple in-process projects create competing claims on the same inputs. A single email, calendar entry, or message may be relevant to more than one project simultaneously.

### 6.2 Project association as interpretation
Project association is a proposed interpretation subject to approval, not a property assigned at ingestion. The system must not assume a single canonical project for any input.

### 6.3 Cross-project signals
Some inputs signal conditions that cut across projects — resource conflicts, schedule dependencies, stakeholder overlap. These must be representable as distinct from single-project associations.

### 6.4 Rule coherence requirement
As rules accumulate, rules belonging to different projects may conflict. A mechanism for surfacing and resolving such conflicts is required. The mechanism is not yet specified (see Section 9).

---

## 7. Approved Case Record

### 7.1 Need
A persistent, replayable record of what was decided, under what conditions, by whom, and why — at the level of individual cases.

### 7.2 Record components
An approved case record must capture:
- The inputs present at decision time
- The proposed interpretation and action that were approved
- The identity of the approving agent (human)
- The rationale, where articulated
- The timestamp of approval

### 7.3 Unanticipated exception constraint
Some decisions will fall outside any existing rule or precedent frame entirely. In these cases, the rationale is not "why was this exception permitted" but "why does this now constitute a precedent." The record structure must accommodate this distinction.

### 7.4 Rationale elicitation
For genuinely novel decisions — those outside any existing frame — the "why" cannot be reconstructed from the trace alone. It requires explicit articulation by the human at decision time. The system must support but cannot guarantee rationale capture for novel cases.

---

## 8. Personal Judgment Model

### 8.1 Nature of what is being learned
The system is learning the individual's judgment, not generic preferences. Judgment involves reasoning structure across diverse and ambiguous situations, not just ranked options or weighted factors.

### 8.2 Non-transferability
The accumulated case library and emergent rules are a portrait of one person's working mind in their specific project context. They are not transferable to another individual and should not be treated as generalizable policy.

### 8.3 Case library as primary asset
Before rules are explicitly extracted, the case library is itself the rules. The system reasons by case analogy in the absence of explicit rules. This is the expected operating mode during Phase 1 and Phase 2.

---

## 9. Open Problems (Not Yet Specified)

The following needs have been identified but not yet resolved:

| Need | Description |
|---|---|
| Similarity function | How the system determines that a new case resembles prior cases sufficiently to surface them as relevant precedent |
| Threshold mechanism | What case density or pattern strength triggers a candidate rule proposal to the human |
| Coherence mechanism | How conflicts between emergent rules across projects are surfaced and resolved |
| Persistence and linking | How the four layers are stored and linked such that provenance is maintained and cases are retrievable |
| Interaction model | How proposals are presented to the human and how approvals are captured with enough structure to be useful as records |
