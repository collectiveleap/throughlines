# Contributing to throughlines

Thanks for your interest. This repo is primarily a personal specification and writing project, but contributions, corrections, and conversations are welcome.

## What this repo is

*throughlines* is a specs-and-writing project exploring durable, regenerable personal architecture. The canonical artifacts here are ideas and specifications; any code that appears (in this repo or downstream) is an implementation of those specs, not the other way around.

## Ways to contribute

- **Issues** — for corrections, questions, disagreements, missing references, or suggestions for new directions. Disagreement is especially welcome; this work is meant to be argued with.
- **Pull requests** — for typos, clarifications, broken links, or additions where there's already shared direction via a prior issue or discussion.
- **Discussions** — for more open-ended exploration that doesn't fit an issue.

For anything substantial (a new section, a reframing, a significant addition to intellectual debts), please open an issue first so we can talk about fit before you invest the effort.

## Licensing of contributions

By opening a pull request or submitting content for inclusion, you agree that your contribution is licensed under [CC BY-SA 4.0](https://creativecommons.org/licenses/by-sa/4.0/), the same license as the rest of the repo. This is the standard inbound-equals-outbound model: you keep authorship; the work stays in the copyleft commons.

If you're contributing material that draws on or adapts a third-party source, please flag it in your PR so it can be added to `ATTRIBUTION.md`.

## Attribution

Contributors to accepted PRs will be credited in `ATTRIBUTION.md` under a Contributors section. If you'd prefer a different form of credit (pseudonym, no credit, specific affiliation), say so in the PR.

## Use of AI tools

This project is drafted in dialogue with AI tools (see `ATTRIBUTION.md`). Contributors are free to use AI tools in their own work on this repo. Two asks:

- Treat AI output as a starting point, not a finished contribution. Editorial judgment is yours.
- If an AI tool materially shaped a contribution, a one-line note in the PR description is appreciated — not as a gatekeeping requirement, just for honest provenance.

## Style and voice

The writing here aims for a specific register: precise, concrete, willing to commit to a position. A few conventions:

- Prefer plain language over jargon. Introduce terms of art explicitly when they're needed.
- Name intellectual debts rather than absorbing them silently.
- When a claim is contested or uncertain, say so. Hedge honestly, not reflexively.
- Short paragraphs. Active voice. Prose over bullets where the ideas have connective tissue.

Contributions don't need to match the primary author's voice, but they should be legible alongside it.

## Scope

This is a focused project with a specific through-line. Contributions that broaden scope significantly — general-interest essays, unrelated topics, tangential tooling — may be declined even if well-executed. This isn't a judgment on the work; it's about keeping the repo coherent.

## Code of conduct

Be decent. Disagree with ideas, not people. Bad-faith engagement, harassment, or dismissiveness toward contributors will result in the conversation being closed. This is a small project; the bar for tone is the bar for a good dinner-party argument.

## Questions

Open a discussion or issue. For anything that shouldn't be public, the primary author's contact information is available via their GitHub profile.
